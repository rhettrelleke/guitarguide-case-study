# Diagrams

Use this folder for a very small number of public-facing diagrams.

Recommended contents:
- one product workflow diagram
- one optional capability-level architecture diagram

Guidelines:
- describe capabilities, not systems
- keep labels short and plain
- avoid data flow, component maps, and internal naming
- remove any diagram that starts to feel like a blueprint

If a diagram would help someone sketch the private codebase with confidence, it does not belong here.
